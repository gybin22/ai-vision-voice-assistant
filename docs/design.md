# 设计文档：AI 视觉语音对话助手

## 1. 项目背景与目标

题目要求开发一款 AI 视觉语音对话应用，并综合考虑：

1. 视觉内容理解准确性；
2. 语音交互自然度与流畅性；
3. 端云协同下的运营成本控制策略。

本项目最终实现的是一个接近“视频通话式交互”的 AI 助手。用户在浏览器中授权摄像头和麦克风后，可以直接通过语音与 AI 对话。前端持续维护最近约 15 秒的视频抽帧，用户停顿后自动提交问题，后端把最近视觉帧和用户问题发送给多模态模型，由模型结合视觉内容和语义意图生成回答。前端展示答案并进行语音播报。

项目不仅实现了基础 AI 调用，还加入了用户登录、平台 Tokens 充值、按模型 usage 计算扣费、请求日志、会话历史和隐私控制，形成一个可演示的商业化 MVP。

---

## 2. 设计原则

### 2.1 视觉准确性优先于单帧截图

早期方案只在用户点击发送时截取当前帧，但这只能回答“现在是什么”，难以回答“刚才发生了什么”。最终方案改为：

- 前端每秒采样 1 帧；
- 最多保留最近 15 帧；
- 每次提问时上传最近 15 秒视觉上下文；
- 发送瞬间补采当前帧；
- 后端告诉模型这些图片是按时间顺序排列的视频抽帧。

这样可以同时支持当前状态识别和短时动作变化理解。

### 2.2 交互体验优先于“每次点发送”

用户不应该每次都点击发送按钮。最终方案加入实时语音模式：

- 点击一次“开始实时对话”；
- 浏览器持续识别用户语音；
- 检测到约 1 秒停顿后自动提交；
- AI 播报期间暂停语音识别，避免回声误识别；
- 播报结束后继续监听下一句。

同时保留手动输入作为浏览器兼容性和复杂问题输入的兜底。

### 2.3 端云协同控制成本

多模态模型成本主要来自：图片输入、文本输入、输出 tokens、高频调用和重复请求。项目采用端云协同：

- 前端控制帧率、帧数和图片压缩；
- 后端限制图片大小、总图片大小、问题长度和输出 tokens；
- 后端限流、限额、缓存；
- 请求成功后按真实 usage 结算平台 Tokens；
- 请求失败不扣费；
- 本地开发使用 Mock 模型，避免开发阶段产生云端成本。

## 5. 核心业务流程

### 5.1 实时语音对话流程

```text
用户点击“开始实时对话”
 → 前端请求摄像头与麦克风权限
 → 摄像头预览启动
 → 最近 15 秒视觉帧缓冲开始采样
 → Web Speech API 连续识别用户语音
 → 检测到约 1 秒停顿
 → 自动提交语音文本
 → 前端上传最近视觉帧 + 当前帧
 → 后端鉴权、限流、余额检查
 → 后端调用多模态模型
 → 后端根据 usage 计算扣费并写入日志
 → 后端保存文本会话历史
 → 前端展示答案并语音播报
 → 播报结束后恢复监听
```

### 5.2 手动输入流程

```text
用户切换到手动输入
 → 输入问题
 → 点击发送
 → 如摄像头未启动，系统自动准备视觉上下文
 → 上传最近 15 秒视觉帧
 → 后端处理流程与实时模式一致
```

### 5.3 Token 结算流程

```text
请求进入后端
 → 校验用户登录态
 → 校验 Tokens 起始余额是否足够
 → 校验图片数量、图片总大小、问题长度、限频
 → 调用模型
 → 解析 input_tokens / output_tokens / total_tokens
 → 计算平台 charged_tokens
 → 扣减用户账户余额
 → 写入 token_transactions
 → 写入 ai_request_logs
 → 返回回答和扣费结果
```

如果模型调用失败：

```text
记录 FAILED 请求日志
 → charged_tokens = 0
 → 不扣减用户 Tokens
 → 返回错误信息
```

## 6. 视觉理解设计

### 6.1 为什么选择最近 15 秒抽帧

如果只上传单张当前帧，模型无法回答：

```text
我刚才做了什么？
刚刚发生了什么变化？
我是不是把东西拿起来了？
```

如果直接上传视频流，则工程复杂度和成本较高。因此最终采用折中方案：

```text
1fps × 15 秒 = 最多 15 张图片
```

该方案成本可控，同时能覆盖大多数 demo 场景中的短动作理解。

### 6.2 当前状态与动作变化的处理

Prompt 中明确告诉模型：

- 图片按时间从早到晚排列；
- 最后一张是用户发送问题瞬间的当前帧；
- 当前状态类问题优先看最后一张；
- “刚才”“过程”“动作”“变化”类问题按时间顺序理解。

### 6.3 避免机械视觉回答

Prompt 要求模型：

- 不要每次都说“根据图片”“画面中可以看到”；
- 问题不需要视觉时就正常聊天；
- 不要逐帧描述；
- 不要在回答中暴露“第几帧”“关键帧”“抽帧”等内部实现；
- 回答保持简短自然，适合语音播报。

## 8. 最终实现与原始计划差异

### 8.1 超出原始 MVP 的实现

最终项目相比最初 MVP 增加了：

- 实时语音自动提交；
- 最近 15 秒视觉滑动窗口；
- 用户注册登录和 JWT 鉴权；
- 平台 Tokens 充值和扣费；
- 模型 usage 成本核算；
- 会话历史；
- qwen3-vl-plus 真实模型配置；
- 不保存图片的隐私设计。

### 8.2 未实现的能力

- 真正 WebRTC 双工实时音视频流；
- 后端 SSE/流式输出；
- 真实支付回调；
- 管理后台；
- 长期记忆或向量数据库；
- 端侧视觉模型；
- 多模型动态路由。

这些能力不是当前 MVP 的必要条件，主要是后续产品化方向。
