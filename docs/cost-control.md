# 运营成本控制策略

## 1. 成本来源分析

本项目的主要成本来自以下几个环节：

| 成本来源 | 说明 |
|---|---|
| 多模态模型输入 tokens | 文本 prompt、历史消息、图片视觉 token |
| 多模态模型输出 tokens | AI 回答越长，输出成本越高 |
| 图片上传带宽 | 多帧图片上传会增加网络和后端处理成本 |
| 高频请求 | 实时语音容易因重复触发造成多次调用 |
| 开发测试 | 开发阶段频繁请求真实模型会产生无效成本 |
| 错误重试 | 401、超时、网关异常等可能导致重复调用 |
| 存储成本 | 如果保存图片或视频，会产生隐私和存储成本 |

因此成本控制需要同时覆盖前端、后端、模型调用、计费和数据存储。

---

## 2. 想到的成本控制技巧

设计阶段考虑过以下策略：

| 编号 | 策略 | 说明 | 是否采用 |
|---|---|---|---|
| C-01 | 不上传连续视频流 | 避免 WebRTC/视频流持续消耗模型资源 | 已采用 |
| C-02 | 只截当前帧 | 最低成本，但无法理解动作变化 | 早期采用，最终升级 |
| C-03 | 最近 15 秒低频抽帧 | 用 1fps 抽帧替代视频流 | 已采用 |
| C-04 | 前端图片压缩 | 在浏览器端降低图片尺寸和质量 | 已采用 |
| C-05 | 限制最大帧数 | 防止一次请求上传过多图片 | 已采用 |
| C-06 | 限制单张图片大小 | 防止大图导致视觉 token 和带宽过高 | 已采用 |
| C-07 | 限制图片总大小 | 控制 multipart 请求体大小 | 已采用 |
| C-08 | 限制问题长度 | 防止长 prompt 滥用 | 已采用 |
| C-09 | 限制模型输出 tokens | 防止回答过长 | 已采用 |
| C-10 | 限制请求频率 | 防止实时语音重复触发 | 已采用 |
| C-11 | 单会话每日配额 | 防止单 session 滥用 | 已采用 |
| C-12 | 单 IP 每日配额 | 防止匿名维度攻击 | 已采用 |
| C-13 | 短上下文 | 只保留最近几轮，不无限拼接历史 | 已采用 |
| C-14 | 请求缓存 | 相同问题和相同视觉上下文命中缓存 | 已采用 |
| C-15 | 浏览器 ASR/TTS | 使用浏览器能力，减少云端语音成本 | 已采用 |
| C-16 | Mock 模型 | 本地开发不调用真实模型 | 已采用 |
| C-17 | 用户 Tokens 余额校验 | 用户余额不足时拒绝调用模型 | 已采用 |
| C-18 | 按 usage 扣费 | 根据模型真实 input/output tokens 扣费 | 已采用 |
| C-19 | 失败不扣费 | 模型失败时不扣用户 Tokens | 已采用 |
| C-20 | 不保存图片 | 降低存储成本和隐私风险 | 已采用 |
| C-21 | 端侧小模型预筛选 | 先判断是否需要视觉 | 未采用 |
| C-22 | 向量数据库语义缓存 | 缓存相似问题和视觉摘要 | 未采用 |
| C-23 | 多模型动态路由 | 简单问题走便宜模型，复杂问题走多模态模型 | 未采用 |
| C-24 | 服务端流式输出 | 降低感知延迟，但不一定降低模型成本 | 未采用 |
| C-25 | 真实支付风控 | 防止恶意充值和刷量 | 未采用，当前为模拟充值 |

## 3. 实际采用的成本控制策略

### 3.1 前端控制

#### 3.1.1 最近 15 秒滑动窗口

前端不上传连续视频流，而是：

```text
每 1 秒采样 1 帧
最多保留最近 15 帧
发送时补采当前帧
```

优势：

- 能理解短时间动作变化；
- 成本远低于实时视频流；
- 请求结构仍是普通 multipart，工程复杂度低。

#### 3.1.2 图片压缩

前端通过 `canvas` 将视频帧压缩为 JPEG，并控制长边和质量。压缩后再上传，减少：

- 请求体积；
- 网络传输耗时；
- 后端内存占用；
- 视觉 token 成本。

#### 3.1.3 实时语音防重复提交

实时语音模式中，前端只在用户停顿约 1 秒后提交，并在 AI `thinking/speaking` 状态下避免继续提交，降低连续语音误触发导致的重复模型调用。

#### 3.1.4 浏览器端 ASR/TTS

语音识别和语音播报优先使用浏览器内置能力：

- Web Speech API；
- SpeechSynthesis。

这样不需要额外调用云端 ASR 或 TTS 服务。

---

### 3.2 后端控制

后端配置位于 `application.yml` 和 `application-qwen.yml`：

```yaml
assistant:
  cost:
    max-image-bytes: 524288
    max-frame-count: 15
    max-total-image-bytes: 3145728
    max-question-length: 500
    max-output-tokens: 500
    min-session-interval-ms: 10000
    max-session-requests-per-day: 30
    max-ip-requests-per-day: 100
    cache-ttl-seconds: 600
    history-rounds: 3
```

#### 3.2.1 单张图片限制

默认单张图片最大 512KB。

如果单张图过大，直接拒绝请求，避免模型成本不可控。

#### 3.2.2 总图片大小限制

默认一次请求图片总大小最大约 3MB。

该限制防止用户上传大量或超大图片导致网络、内存和模型视觉输入成本飙升。

#### 3.2.3 帧数限制

默认最多 15 帧。

即使前端异常上传更多帧，后端也会拦截。

#### 3.2.4 问题长度限制

默认问题最长 500 字，防止超长文本 prompt。

#### 3.2.5 输出 tokens 限制

默认最多 500 tokens。结合 Prompt 要求 1 到 2 句话，实际输出通常更短。

#### 3.2.6 请求限频和配额

后端按 session 和 IP 进行控制：

- 单 session 最小请求间隔；
- 单 session 每日请求上限；
- 单 IP 每日请求上限。

#### 3.2.7 短上下文

只读取最近 3 轮上下文，避免历史消息无限膨胀。

#### 3.2.8 缓存

使用问题、视觉摘要和图片 hash 生成缓存 key。

相同 session 中同样问题和相同视觉上下文可以命中缓存，减少重复模型调用。

---

### 3.3 模型调用控制

#### 3.3.1 Mock 与真实模型分离

本地开发默认使用 Mock：

```yaml
assistant:
  model:
    provider: mock
```

真实演示或部署时再启用 qwen profile：

```bash
SPRING_PROFILES_ACTIVE=qwen
DASHSCOPE_API_KEY=sk-xxx
```

这样可以避免开发阶段无意义消耗模型费用。

#### 3.3.2 OpenAI-Compatible 适配

后端通过 `MultimodalModelClient` 抽象模型供应商。当前包括：

- `MockVisionClient`；
- `OpenAiCompatibleVisionClient`。

后续可扩展其它模型，而不会影响前端和业务逻辑。

#### 3.3.3 Prompt 控制输出长度

Prompt 要求：

- 直接回答；
- 通常 1 到 2 句话；
- 不逐帧描述；
- 不额外展开背景；
- 不强行描述画面。

这既提升自然度，也减少输出 tokens。

---

### 3.4 平台 Tokens 计费控制

#### 3.4.1 请求前余额检查

每次 AI 视频对话前先检查余额，余额不足则不调用模型。

默认至少需要满足：

```text
max(min-start-balance-tokens, min-visual-charge-tokens)
```

#### 3.4.2 根据真实 usage 扣费

模型调用成功后，后端读取：

```text
input_tokens
output_tokens
total_tokens
```

并计算平台 Tokens：

```text
platform_tokens_used = max(
  min_visual_charge,
  input_tokens × input_multiplier + output_tokens × output_multiplier
)
```

默认：

```yaml
input-token-multiplier: 1
output-token-multiplier: 4
min-visual-charge-tokens: 5000
```

这样可以把真实模型成本、平台收入和用户余额统一起来。

#### 3.4.3 记录真实成本与毛利

每次请求会记录：

- input_tokens；
- output_tokens；
- total_tokens；
- charged_tokens；
- token_unit_price_yuan；
- revenue_amount_yuan；
- provider_cost_amount_yuan；
- gross_profit_amount_yuan；
- frame_count；
- image_total_bytes；
- latency_ms；
- status。

这些数据保存在 `ai_request_logs` 中，后续可以直接用于运营分析。

#### 3.4.4 模型失败不扣费

如果模型调用失败，例如认证异常、网络异常或超时，系统会记录失败日志，但不扣减用户 Tokens。

---

### 3.5 存储成本与隐私控制

系统不保存用户上传的图片帧。数据库只保存：

- 用户文字问题；
- AI 文本回答；
- 模型名；
- token 用量；
- 扣减 Tokens；
- 会话时间。

这降低了存储成本，也减少了摄像头图片带来的隐私风险。

---

## 4. 未采用策略及原因

### 4.1 没有直接传视频流

未采用 WebRTC 或连续视频流上传。原因：

- 工程复杂度高；
- 模型成本不可控；
- 服务端需要维护长连接和状态；
- 当前 demo 场景中 15 秒抽帧已足够覆盖核心能力。

### 4.2 没有接入真实支付

当前充值是模拟流程。原因：

- Demo 更关注计费模型和账户流水；
- 真实支付需要支付平台资质、回调验签和风控；
- 数据库设计已保留扩展空间。

### 4.3 没有使用向量数据库

向量数据库可用于语义缓存，但当前问题规模较小，且视觉内容变化快。MVP 阶段先采用 hash 缓存和限流更直接。

### 4.4 没有使用端侧视觉模型

端侧模型可先判断是否需要上传视觉内容，但会增加前端包体、兼容性和性能问题。当前采用轻量抽帧和压缩方式更稳定。

---

## 5. 成本控制结论

最终采用的成本控制方案可以概括为：

```text
前端少传：低频抽帧、限制帧数、图片压缩
后端严控：限大小、限频、限额、短上下文、缓存
模型节制：短回答、限制输出、Mock/真实模型分离
商业闭环：余额校验、usage 扣费、失败不扣费、成本核算
隐私节省：不保存图片，只保存文本和元数据
```

该方案在保证视觉理解能力的同时，把模型调用成本、存储成本、网络成本和误触发成本控制在可接受范围内。
